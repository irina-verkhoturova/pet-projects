# RFM-сегментация пользователей стримингового сервиса

**Описание.** Полный пайплайн поведенческой сегментации на сыром event-логе сессий: профилирование качества, чистка, агрегация в RFM-витрину, фильтр ботов, EDA, скоринг и кластеризация, сегменты и продуктовые гипотезы. RFM адаптирован под стриминг: Recency это давность последней сессии, Frequency это число сессий, Monetary это минуты просмотра.

**Навыки.** RFM-сегментация, обработка сырого event-лога, профилирование и чистка данных, фильтрация ботов и выбросов, винзоризация по 99-му перцентилю, кластеризация (KMeans), детерминированное именование сегментов, корреляционный анализ, генерация продуктовых гипотез, дизайн метрики удержания. Агрегация в витрину приведена и в виде эквивалентного SQL-запроса на PostgreSQL.

**Стек.** Python, pandas, numpy, scipy, scikit-learn, seaborn, matplotlib.

**Результат.** Из сырого лога убраны дубликаты событий, около 2% строк без `user_id` и около 3% с непарсящимися датами, время просмотра винзоризовано по 99-му перцентилю. Отдельно отфильтрованы боты (более 150 сессий при нулевом просмотре), иначе они искажали бы сегменты. Выделено пять устойчивых сегментов. Ключевая находка это сегмент «спящие»: высокая частота входов при просмотре около нуля. По login-retention они выглядят здоровыми, поэтому стандартная метрика удержания маскирует проблему, это скрытый отток. Частота сессий сильно коррелирует с retention (r около 0,9), но частота не равна потреблению. Главная рекомендация: ввести consumption-retention, долю недель с реальным просмотром, как основную метрику удержания, и проверить на «спящих» персонализацию главного экрана через A/B.

**Данные.** Синтетический сырой лог сессий `streaming_events_raw.csv`, специально «грязный» (дубликаты, пропуски, кривые даты, боты), чтобы пайплайн чистки был реалистичным. Данные не конфиденциальные, поэтому файл можно положить прямо в папку проекта рядом с ноутбуком, и тогда ноутбук запустится сразу.
